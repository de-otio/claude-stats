### Detailed Steps

1. **User runs `claude-stats sync --setup`**
2. CLI prompts for email, calls Cognito `InitiateAuth` (CUSTOM_AUTH)
3. Cognito triggers magic link flow (see [02-authentication.md](02-authentication.md))
4. CLI polls for completion or accepts pasted verification URL
   - Polling: CLI starts a local HTTP listener on a random port. The magic link URL includes a `redirect_uri=http://localhost:{port}/callback` parameter. When the user clicks the link in their browser, the SPA `/auth/verify` page completes the Cognito challenge and redirects to the CLI's local listener with the tokens.
   - Paste fallback: user copies the magic link URL, CLI extracts the token and calls `RespondToAuthChallenge` directly.
5. CLI receives Cognito tokens (access, ID, refresh) and extracts `userId` from the `sub` claim
6. CLI generates `userSalt` (32-byte random hex) if first setup, or fetches existing salt from cloud profile
7. CLI reads `~/.claude.json` to detect the current `account_uuid`
8. CLI computes `accountId = HMAC-SHA-256(account_uuid, userSalt).slice(0, 32)`
9. CLI calls `linkAccount` mutation and `updateProfile` mutation
10. CLI stores all config in local SQLite `sync_config` table:

```sql
-- sync_config entries after setup
INSERT OR REPLACE INTO sync_config (key, value) VALUES
  ('appsync_endpoint',     'https://xxx.appsync-api.us-east-1.amazonaws.com/graphql'),
  ('cognito_user_pool_id', 'us-east-1_XXXXXXXXX'),
  ('cognito_client_id',    '1234567890abcdef'),
  ('user_id',              'abc-123'),
  ('user_salt',            'deadbeef...'),
  ('enabled',              'true'),
  ('refresh_token',        'eyJ...');  -- Encrypted at rest (see Token Storage below)
```

### Backend Endpoint Discovery

The CLI needs three values to connect: AppSync endpoint, Cognito User Pool ID, and Cognito Client ID. These are provided via one of:

| Method | When | How |
|--------|------|-----|
| **Well-known URL** (default) | Hosted deployments | CLI fetches `https://{domain}/.well-known/claude-stats.json` from the SPA's CloudFront distribution. The file is generated at build time from SSM parameters (same as SPA config). |
| **Manual entry** | Self-hosted / dev | `claude-stats sync --setup --endpoint https://xxx.appsync-api...` |
| **Environment variables** | CI / Docker | `CLAUDE_STATS_ENDPOINT`, `CLAUDE_STATS_COGNITO_POOL_ID`, `CLAUDE_STATS_COGNITO_CLIENT_ID` |

Deployed to the SPA S3 bucket alongside the frontend:

## Token Storage (CLI)

Cognito tokens are stored in the local SQLite `sync_config` table:

| Token | Storage | Lifetime |
|-------|---------|----------|
| Access token | In-memory only (re-derived from refresh token) | 1 hour |
| Refresh token | SQLite `sync_config`, encrypted with OS keychain key | 30 days |
| ID token | Not stored (only used during setup for `sub` claim) | ‚Äî |

### Refresh Token Encryption

The refresh token is sensitive (grants 30-day access). It is encrypted before storing in SQLite:

| Platform | Key Source |
|----------|-----------|
| macOS | Keychain Services (`security add-generic-password`) ‚Äî key stored in login keychain |
| Linux | `libsecret` via `keytar` if available, otherwise XDG-compliant file with `0600` permissions |

If no OS keychain is available, the refresh token is stored in plaintext in `sync_config` with a warning during setup. This is acceptable for the CLI's threat model (local user's own machine).

### Token Refresh Flow

```
CLI sync command starts
  ‚Üí Read refresh_token from sync_config
  ‚Üí Call Cognito InitiateAuth with REFRESH_TOKEN_AUTH flow
  ‚Üí Receive new access_token (1h TTL)
  ‚Üí Use access_token for all AppSync requests
  ‚Üí If refresh_token is expired ‚Üí prompt user to re-authenticate
```

## VS Code Extension Integration

The extension does not duplicate auth logic. It delegates to the CLI and reads sync state from SQLite.

### Additional Extension Settings

```json
// extension/package.json ‚Äî contributes.configuration
{
  "claude-stats.syncEnabled": {
    "type": "boolean",
    "default": false,
    "description": "Enable cloud sync (requires `claude-stats sync --setup` first)"
  },
  "claude-stats.backendUrl": {
    "type": "string",
    "default": "",
    "description": "Claude Stats backend URL (e.g. https://stats.acme.com). Leave blank to use CLI config."
  }
}
```

### Extension Behavior

1. **Dashboard panels**: The local dashboard (`serve` command) already renders HTML. When `syncEnabled` is true, the dashboard includes additional panels:
   - "Synced across N devices" indicator
   - "Last synced: 2 minutes ago" status
   - Link to open the SPA for team features

2. **Sync trigger**: The extension runs `claude-stats sync --push` after each `collect` cycle (respecting the `autoRefreshSeconds` interval). This is a background process ‚Äî no UI blocking.

3. **Status bar**: When sync is enabled, the status bar shows sync state:
   ```
   Claude Stats: ‚úì Synced | 47 sessions today
   Claude Stats: ‚Üª Syncing... | 47 sessions today
   Claude Stats: ‚úó Sync error (click for details)
   ```

4. **Setup command**: The extension provides a command palette entry:
   ```
   Claude Stats: Connect to Backend
   ```
   This opens a terminal and runs `claude-stats sync --setup`. After setup completes, the extension reads `sync_config` from SQLite and enables sync.

5. **Open team dashboard**: Command palette entry:
   ```
   Claude Stats: Open Team Dashboard
   ```
   Opens the SPA URL (from `sync_config.appsync_endpoint` domain or `backendUrl` setting) in the default browser.

### Extension Config Precedence

```

    ‚Üí falls back to ‚Üí CLAUDE_STATS_URL env var
```

## SPA Connection

The SPA is fully self-contained ‚Äî its config is baked at build time:

```typescript
// src/config.ts (generated at build, see 07-frontend.md)
export default {
  cognitoUserPoolId: "us-east-1_XXXXXXXXX",
  cognitoClientId: "1234567890abcdef",
  appSyncEndpoint: "https://xxx.appsync-api.us-east-1.amazonaws.com/graphql",
} as const;
```

No user configuration needed. The SPA handles its own auth flow (magic link ‚Üí Cognito ‚Üí JWT stored in browser).

## Multi-Account Setup

Users with multiple Claude accounts (work + personal) run setup once per account:

```bash
# First account (currently active in ~/.claude.json)
claude-stats sync --setup
# Label: "Work", share: yes

# Switch Claude accounts, then:
claude-stats sync --setup --link-account
# Label: "Personal", share: no
```

The `--link-account` flag skips authentication (already authenticated) and just links the current `~/.claude.json` account. See [11-account-separation.md](11-account-separation.md) for the full account model.

## Disconnect

```bash
claude-stats sync --disconnect

  This will:
  - Disable cloud sync
  - Remove stored credentials
  - Keep all local data intact

  Proceed? [y/N]: y

  ‚úì Disconnected. Local data preserved.
    Re-run `claude-stats sync --setup` to reconnect.
```

Clears all `sync_config` entries except `user_salt` (preserved so re-linking produces the same `accountId` values). Cloud data is not deleted ‚Äî use `deleteMyAccount` mutation via the SPA for full data purge.

## Environment Variables

All CLI/extension configuration can be overridden via environment variables:

| Variable | Purpose | Example |
|----------|---------|---------|
| `CLAUDE_STATS_URL` | Backend URL for endpoint discovery | `https://stats.acme.com` |
| `CLAUDE_STATS_ENDPOINT` | Direct AppSync endpoint (skips discovery) | `https://xxx.appsync-api...` |
| `CLAUDE_STATS_COGNITO_POOL_ID` | Cognito User Pool ID | `us-east-1_XXXXXXXXX` |
| `CLAUDE_STATS_COGNITO_CLIENT_ID` | Cognito Client ID | `1234567890abcdef` |
| `CLAUDE_STATS_SYNC_ENABLED` | Enable/disable sync | `true` / `false` |

Environment variables take precedence over SQLite `sync_config` and VS Code settings.

## Infrastructure Changes

### FrontendStack

The `BucketDeployment` must include the `.well-known/claude-stats.json` file alongside the SPA build output. The `generate-config.ts` script writes both `src/config.ts` (SPA) and `.well-known/claude-stats.json` (CLI discovery) from the same SSM parameters.

### Cognito Client Configuration

The existing SPA Cognito client already supports `CUSTOM_AUTH`. For CLI auth, the same client is used with one addition:

- **Callback URL**: The SPA client's `CallbackURLs` must include `http://localhost:*/callback` (wildcard port) to support the CLI's local listener redirect flow. In dev, `http://localhost:*` is already allowed. In prod, add the localhost pattern ‚Äî Cognito allows it since the redirect happens in the user's browser, not over the network.

Alternatively, use the paste-the-link fallback exclusively (simpler, no callback URL needed). The local listener approach is better UX but optional.

### SSM Parameters

No new SSM parameters needed. The CLI discovery file reads the same parameters the SPA config uses:
- `/{prefix}/api/graphql-endpoint`
- `/{prefix}/auth/user-pool-id`
- `/{prefix}/auth/spa-client-id`
